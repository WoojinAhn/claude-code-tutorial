# 모듈 6: 개발자 도구

이번 모듈에서는 개발 과정에서 유용한 유틸리티성 커맨드들을 다룹니다.

환경 문제 진단, 터미널 최적화, 테마 변경, 코드 리뷰까지 — 일상적인 개발 워크플로우를 더 편하게 만들어주는 도구들입니다.

---

## 6.1 /doctor

**설명:** `/doctor`는 Claude Code의 건강 상태를 체크합니다. 설정 오류, 인증 문제, 환경 설정 이슈 등 일반적인 문제들을 자동으로 진단해줍니다. 뭔가 안 될 때 가장 먼저 실행해볼 커맨드입니다.

**문법:**
```
/doctor
```

**예시:** `/doctor`를 실행하면 여러 항목을 자동으로 검사합니다:
- 인증 상태 (API 키, 로그인 상태)
- 환경 설정 (Node.js 버전, 필수 의존성)
- 설정 파일 유효성
- 네트워크 연결 상태

문제가 발견되면 해결 방법도 함께 안내해줍니다.

**실습:** 지금 `/doctor`를 실행해보세요. 모든 항목이 정상인지 확인해보세요.

**확인:** 실행해봤나요? 모든 체크가 통과했나요, 아니면 경고가 있었나요? 어떤 결과가 나왔는지 알려주세요.

**팁:** Claude Code가 갑자기 느려지거나, 이상한 에러가 나거나, 도구가 작동하지 않을 때 `/doctor`부터 돌려보세요. 대부분의 환경 문제는 여기서 원인을 찾을 수 있습니다.

---

## 6.2 /terminal-setup

**설명:** `/terminal-setup`은 터미널 환경을 Claude Code에 최적화합니다. Shift+Enter 멀티라인 입력, 키바인딩 설정, 표시 최적화 등을 안내합니다.

**문법:**
```
/terminal-setup
```

**예시:** `/terminal-setup`을 실행하면 현재 터미널 환경에 맞는 최적화 설정을 안내해줍니다:
- Shift+Enter로 줄바꿈 지원 설정
- "Option as Meta Key" 설정 (macOS)
- 터미널별 최적화 안내 (iTerm2, Terminal.app, VS Code 등)

**실습:** 지금 `/terminal-setup`을 실행해보세요. 안내되는 설정을 확인해보세요.

**확인:** 실행해봤나요? 어떤 설정이 안내됐는지 알려주세요. 이미 설정되어 있는 항목도 있을 수 있습니다.

**팁:** Shift+Enter 줄바꿈 설정은 꼭 해두세요. 설정하지 않으면 Enter를 누르는 순간 바로 전송돼서, 긴 프롬프트를 작성하기 어렵습니다. 일회성 설정이지만 매일의 사용감이 크게 달라집니다.

---

## 6.3 /theme

**설명:** `/theme`은 Claude Code의 색상 테마를 변경합니다. 기본 테마가 마음에 안 들거나 터미널 환경에 맞지 않으면, 원하는 테마로 바꿀 수 있습니다.

**문법:**
```
/theme
```

**예시:** `/theme`을 실행하면 테마 선택기가 열립니다. 사용 가능한 테마를 탐색하고 선택할 수 있습니다. 테마 선택기 안에서 `Ctrl+T`를 누르면 코드 블록의 구문 강조 표시를 토글할 수 있습니다.

**실습:** 지금 `/theme`을 실행해보세요. 마음에 드는 테마를 골라보세요. 언제든 다시 바꿀 수 있습니다.

**확인:** 실행해봤나요? 마음에 드는 테마를 찾았나요? 변경은 즉시 적용됩니다.

**팁:** 작업 환경의 조명이 바뀌면 (밝은 사무실 vs 어두운 방) 테마도 맞춰 바꾸면 눈이 편합니다. 밝은 터미널에는 라이트 테마, 어두운 터미널에는 다크 테마가 잘 맞습니다.

---

## 6.4 /review

**설명:** `/review`는 코드 리뷰를 시작합니다. Claude가 현재 작업 디렉토리의 Git 변경사항을 분석해서 잠재적 문제, 코드 스타일, 개선점을 짚어줍니다.

**문법:**
```
/review
```

**예시:** `/review`를 실행하면 Claude가 현재 변경사항을 분석합니다:
- 잠재적 버그나 에지 케이스 발견
- 코드 품질 및 가독성 피드백
- 보안 관련 주의사항
- 개선 제안

"보안 위주로 리뷰해줘"나 "에러 핸들링 위주로 봐줘"처럼 초점을 지정할 수도 있습니다.

**실습:** Git 리포지토리 안에 있다면 지금 `/review`를 실행해보세요. 리포지토리가 아니라면, 나중에 실제 프로젝트에서 해보세요.

**확인:** 실행해봤나요? 어떤 피드백이 나왔는지 알려주세요. 리포지토리가 아니어서 실행하지 못했다면 그것도 괜찮습니다.

**팁:** PR을 올리기 전에 `/review`를 한번 돌려보는 습관을 들이면 좋습니다. 명백한 실수를 사람 리뷰어가 보기 전에 잡을 수 있습니다. 다만 Claude의 리뷰는 사람 리뷰를 대체하는 게 아니라 보완하는 겁니다.
