# 모듈 4: 개발자 도구

이번 모듈에서는 개발 과정에서 유용한 유틸리티성 커맨드들을 다룹니다.

환경 문제 진단, 에디터 키바인딩, 터미널 최적화, 코드 리뷰까지 — 일상적인 개발 워크플로우를 더 편하게 만들어주는 도구들입니다.

---

## 4.1 /doctor

**설명:** `/doctor`는 Claude Code의 건강 상태를 체크합니다. 설정 오류, 인증 문제, 환경 설정 이슈 등 일반적인 문제들을 자동으로 진단해줍니다. 뭔가 안 될 때 가장 먼저 실행해볼 커맨드입니다.

**문법:**
```
/doctor
```

**예시:** `/doctor`를 실행하면 여러 항목을 자동으로 검사합니다:
- 인증 상태 (API 키, 로그인 상태)
- 환경 설정 (Node.js 버전, 필수 의존성)
- 설정 파일 유효성
- 네트워크 연결 상태

문제가 발견되면 해결 방법도 함께 안내해줍니다.

**실습:** 지금 `/doctor`를 실행해보세요. 모든 항목이 정상인지 확인해보세요.

**확인:** 실행해봤나요? 모든 체크가 통과했나요, 아니면 경고가 있었나요? 어떤 결과가 나왔는지 알려주세요.

**팁:** Claude Code가 갑자기 느려지거나, 이상한 에러가 나거나, 도구가 작동하지 않을 때 `/doctor`부터 돌려보세요. 대부분의 환경 문제는 여기서 원인을 찾을 수 있습니다.

---

## 4.2 /vim

**설명:** `/vim`은 입력 영역에서 vim 스타일 키바인딩을 켜거나 끕니다. vim 사용자라면 익숙한 모달 편집(Normal/Insert 모드)을 Claude Code 프롬프트에서도 쓸 수 있습니다.

**문법:**
```
/vim
```

**예시:** `/vim`을 실행하면 vim 모드가 토글됩니다:
- **켜짐:** `Esc`로 Normal 모드, `i`로 Insert 모드. `dd`, `yy`, `p` 같은 기본적인 vim 동작이 됩니다.
- **꺼짐:** 일반적인 텍스트 입력 방식으로 돌아갑니다.

**실습:** 지금 `/vim`을 실행해보세요. vim 모드를 켜본 다음, 한 번 더 실행해서 꺼보세요.

**확인:** 실행해봤나요? vim 모드가 켜졌다 꺼졌나요? 어떤 결과가 나왔는지 알려주세요.

**팁:** vim에 익숙하지 않다면 굳이 켤 필요 없습니다. 하지만 vim 사용자라면 프롬프트에서 긴 명령어를 편집할 때 상당히 편합니다. 실수로 켰다가 입력이 안 되면 당황하지 말고 `/vim`으로 다시 끄면 됩니다.

---

## 4.3 /terminal-setup

**설명:** `/terminal-setup`은 터미널 환경을 Claude Code에 최적화합니다. 더 나은 렌더링, 유니코드 지원, 색상 표시 등을 위한 터미널 설정을 안내합니다.

**문법:**
```
/terminal-setup
```

**예시:** `/terminal-setup`을 실행하면 현재 터미널 환경에 맞는 최적화 설정을 안내해줍니다:
- 코드 블록이 제대로 표시되도록 폰트 설정
- 유니코드 문자 렌더링 개선
- 터미널 색상 테마 최적화
- Shift+Enter로 줄바꿈 지원 설정

**실습:** 지금 `/terminal-setup`을 실행해보세요. 안내되는 설정을 확인해보세요.

**확인:** 실행해봤나요? 어떤 설정이 안내됐는지 알려주세요. 이미 설정되어 있는 항목도 있을 수 있습니다.

**팁:** Shift+Enter 줄바꿈 설정은 꼭 해두세요. Claude Code에서 여러 줄의 프롬프트를 작성할 때 매우 유용합니다. 설정하지 않으면 Enter를 누르는 순간 바로 전송돼서, 긴 지시를 작성하기 어렵습니다.

---

## 4.4 /review

**설명:** `/review`는 코드 리뷰를 시작합니다. PR(Pull Request)이나 작업 중인 변경사항에 대해 Claude가 리뷰를 해줍니다. 버그, 보안 이슈, 코드 스타일, 개선점 등을 짚어줍니다.

**문법:**
```
/review
```

**예시:** `/review`를 실행하면 Claude가 현재 작업 디렉토리의 Git 변경사항을 분석합니다:
- 커밋되지 않은 변경사항 분석
- 잠재적 버그나 에지 케이스 발견
- 코드 품질 및 가독성 피드백
- 보안 관련 주의사항

Git 리포지토리에서 실행해야 의미 있는 결과가 나옵니다.

**실습:** Git 리포지토리 안에 있다면 지금 `/review`를 실행해보세요. 리포지토리가 아니라면, 나중에 실제 프로젝트에서 해보세요.

**확인:** 실행해봤나요? 어떤 피드백이 나왔는지 알려주세요. 리포지토리가 아니어서 실행하지 못했다면 그것도 괜찮습니다.

**팁:** PR을 올리기 전에 `/review`를 한번 돌려보는 습관을 들이면 좋습니다. 사람 리뷰어가 잡아주기 전에 명백한 실수들을 미리 잡을 수 있습니다. 다만, Claude의 리뷰를 맹신하지는 마세요 — 사람 리뷰를 대체하는 게 아니라 보완하는 겁니다.
